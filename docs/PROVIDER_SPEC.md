# Спецификация провайдера: данные, возможности, контракт API

Документ для **владельца или оператора бэкенда** (прокси к Max, отдельный хостинг API и т.п.). По этим сведениям можно оценить совместимость с MaxNotify и подготовить Pull Request в репозиторий интеграции.

Связанный документ: [внедрение в код](PROVIDER_PULL_REQUEST.md).

---

## Соответствие ответов официальному API

**Обязательно:** для каждого используемого метода и пути ответ провайдера **полностью совпадает по контракту** с ответом **официального Max API** ([документация platform-api](https://dev.max.ru/docs-api)) при той же версии `v=` и тех же входных условиях.

Имеется в виду:

- те же **коды HTTP** и та же **логика успеха/ошибки**;
- те же **структура и имена полей** в JSON (включая вложенность и типы значений), что описаны для официального API;
- для **приёма обновлений** — элементы списка `updates` и сырой payload вебхука в форме **официального** `Update`, без альтернативных схем.

Обобщённые или «похожие» ответы, нормализация на стороне провайдера под видом другого JSON **не подходят**: интеграция и общий код рассчитывают на **тот же контракт тела ответа**, что у platform-api (структура и семантика полей как в официальной документации). Отдельно документируйте только **ограничения по объёму и частоте** (лимиты, таймауты), не меняющие формат ответа.

## 1. Идентификация и конечные точки

| Что передать | Зачем в интеграции |
|----------------|-------------------|
| **Строковый тип интеграции** (`integration_type`), например `my_provider` | Сохраняется в `ConfigEntry.data["integration_type"]`, ключ в реестре провайдеров. Стабильный идентификатор, без пробелов; лучше латиница и подчёркивания. |
| **Короткое человекочитаемое имя** (label) | Подписи в UI, плейсхолдер `{provider}` в ошибках. |
| **Базовый URL API** | Без завершающего `/` (если иное не оговорено). От него собираются пути вроде `/messages`, `/uploads`, `/updates`. |
| **Версия API** для query-параметра `v=` | Как у platform-api Max; участвует в URL исходящих запросов. |

При отличии путей от «канонических» (`/messages`, `/uploads`, `/updates`, подписки webhook и т.д.) нужен **явный маппинг**: какой ваш URL соответствует какому сценарию в интеграции.

## 2. Аутентификация и токен

| Что передать | Зачем |
|----------------|------|
| **Формат заголовка `Authorization`** | В общем коде значение токена из настроек передаётся в HTTP как заголовок `Authorization` **ровно с тем значением, которое ввёл пользователь** — префикс `Bearer ` интеграция **не добавляет сама**. Если API ожидает вид `Bearer` + пробел + токен, это должно быть в инструкции пользователю или в логике конкретного провайдера в коде. |
| **Способ проверки токена** | Эндпоинт и метод (аналог `GET /me` или эквивалент), ожидаемые коды ответа при успехе и ошибке. |
| **Ожидаемая длина токена** (число символов) или «любая ненулевая строка» | Если задана фиксированная длина — мастер настройки и переводы могут валидировать ввод (`access_token_length` у провайдера). `null` — без проверки длины в UI. |
| **Модель срока жизни** | В документации интеграции принимается сценарий **долгоживущего** токена без обязательного описания TTL и ротации; если у вас иначе — опишите отдельно для пользователей. |
| **Несколько записей HA с одним токеном** | Разрешено или нет; конфликтуют ли режимы приёма (webhook vs polling) между записями. Это влияет на флаги `allow_multiple_config_entries_same_token`, `shares_platform_bot_token_pool` и методы вроде `iter_config_entries_sharing_token`. |

---

## 3. Идентификаторы получателя (Max-совместимая модель)

- **Положительный** числовой ID — личный чат (`user_id`).
- **Отрицательный** — групповой чат (`chat_id`).

Если групповые чаты **не** поддерживаются, это обязательно отражается во **флаге возможностей** (`supports_group_chats: false`): иначе пользователь получит ошибку только на этапе вызова API.

## 4. Возможности бэкенда (матрица для `IntegrationCapabilities`)

Для каждой строки укажите **да / нет** и при «да» — кратко, как это выглядит в API (метод, путь, ограничения).

| Поле (логика в коде) | Смысл |
|----------------------|--------|
| `supports_group_chats` | Отрицательный `recipient_id`. |
| `supports_inline_keyboard` | Inline-кнопки в исходящих сообщениях и при редактировании. |
| `supports_delete_message` | Удаление сообщения по `message_id`. |
| `supports_edit_message` | Редактирование текста/клавиатуры. |
| `supports_send_photo` | Цепочка загрузки + сообщение с вложением «фото». |
| `supports_send_document` | Аналогично для файла. |
| `supports_send_video` | Аналогично для видео. |
| `supports_bot_commands` | Синхронизация slash-команд с бэкендом (как у официального API). |
| `max_client_upload_bytes` | Верхняя граница размера тела вложения для **предпроверки** на стороне HA (или `null`, если не ограничиваете на клиенте). |
| `supports_receive_polling` | Периодический опрос очереди обновлений без long-hold. |
| `supports_receive_long_polling` | Long polling к `/updates` (или эквиваленту) с `marker` и таймаутом. |
| `supports_receive_webhook` | Входящие POST на URL HA; для платформы Max — модель подписок и секрет заголовка. |

Дополнительно в **конструкторе провайдера** (не в `IntegrationCapabilities`) задаются, среди прочего:

- `supports_receive_polling` / `supports_receive_long_polling` на уровне класса (дублируют часть семантики вместе с capabilities — см. существующие провайдеры в `registry.py`);
- лимиты polling: `updates_poll_limit`, `updates_interval_default` / `min` / `max`;
- `receive_modes` — какие режимы приёма показывать в мастере;
- `update_types_receive` — какие типы апдейтов запрашивать у `/updates`;
- `max_attachments_per_message_limit` — лимит вложений на одно сообщение на стороне интеграции (или `null`).

Точные имена полей см. в `MaxNotifyIntegrationProvider` в `providers/base.py`.

## 5. Исходящие сообщения (обязательный минимум и опции)

### 5.1. Обязательный минимум (Обязательно)

- Отправка **текстовых** сообщений.
- Тело **запроса** следует контракту официального Max API для соответствующего метода; тело **ответа** — **полностью** как у официального API (см. раздел [«Соответствие ответов официальному API»](#соответствие-ответов-официальному-api) выше).
- Описаны метод, путь, пример успешного и ошибочного ответа **в терминах официальной документации** (без отдельной «своей» схемы JSON).
- Для отправки сообщений **всех типов** (text/photo/document/video, включая кнопки и `notify`) провайдер обязан передавать на серверы MAX (`/messages`) сформированное интеграцией тело запроса **без изменений полей, структуры и значений**; недопустимы «тихие» правки, удаление или переименование полей на стороне провайдера.

### 5.2. Опционально

Для каждой включённой возможности из раздела 4 — отдельно:

- HTTP-метод и URL;
- обязательные и необязательные поля;
- примеры JSON;
- ограничения: размер файла, число вложений, rate limit, таймауты.

Типовые сценарии в интеграции: `POST /messages`, `POST /uploads`, `DELETE`/`PUT` для сообщений — пути должны быть согласованы с официальной схемой или явно замаплены.

## 6. Входящие обновления (приём)

Укажите, что реально доступно:

| Режим | Что описать |
|--------|-------------|
| Только отправка | Ничего дополнительно. |
| Long polling | Параметры запроса, таймаут, поле **marker**, формат списка `updates`. |
| Интервальный polling | Период, лимит выборки, нужен ли pacing между запросами. |
| Webhook | Формат подписи запроса, секрет, совместимость с моделью Max `/subscriptions` (если эмулируете платформу). |

Нужны **примеры JSON** минимум для события сообщения и для callback по кнопке — в **том же виде**, в каком их отдаёт официальный API (объекты `Update`). После обработки интеграция публикует событие Home Assistant `max_notify_received` с полями вроде `update_type`, `config_entry_id`, `timestamp`, `recipient_id`, `text`, `command`, `args`, `callback_data`, `message_id`, `event_id`, `raw_update`.

---

## 7. Дополнительные ограничения и поведение

Провайдер обязан явно документировать ограничения **поверх** официального контракта (формат ответов при этом остаётся как у platform-api):

- лимиты размера и числа файлов;
- особенности очереди polling (TTL, потеря апдейтов);
- ограничения по типам чатов;
- квоты, задержки, отличия **сроков хранения** сообщений на стороне сервиса (если есть), **не** меняющие структуру JSON ответа.

Если фактический ответ на эндпоинте **не** совпадает с официальным API — такой бэкенд **не соответствует** этой спецификации; доработка под иной формат потребует отдельного согласования и, как правило, отдельной логики в коде интеграции.

## 8. Сводная таблица требований к HTTP API

| Требование | Статус |
|------------|--------|
| Запросы и **ответы** по контракту официального Max API (полное соответствие тел ответов и ошибок) | Обязательно |
| Передача токена в `Authorization` в согласованном с вами виде | Обязательно |
| Семантика ID: положительный — личный, отрицательный — группа (или явный отказ от групп) | Обязательно |
| Перечень методов и путей для всех заявленных возможностей | Обязательно |
| Примеры ответов и ошибок | Обязательно |
| Медиа, delete/edit, webhook/polling/long polling | Опционально — только для заявленных флагов |
